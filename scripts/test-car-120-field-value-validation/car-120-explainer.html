CAR-120 · Feature

Chart notes now reject
garbage field values.

Before this PR, saveDraft accepted pain_scale: "seven and a half" against a 0 – 10 numeric scale without complaint. JSONB blobs accumulated garbage that every downstream consumer — PDF, signed-note display, the future signature gate, AI round-trips — had to defend against on its own. This PR closes that gap at the aggregate.

Before · After

A lie the old API used to tell.

Same request, same template, same chart note. The only thing that changed is where the invariant lives.

Before · 200 OK PATCH /chart-notes/:id 
{
  "incomingFieldValues": {
    "pain_scale": "seven and a half",
    "mechanism": "Traumatisque",
    "rom_log":   [{"col_b": 12}]
  },
  "version": 3
}

→ 200 OK, version bumped to 4
  garbage persisted in JSONB
After · 422 Validation PATCH /chart-notes/:id 
{
  "error": {
    "code": "FIELD_VALUE_VALIDATION_ERROR",
    "message": "3 field value(s) failed validation",
    "errors": [
      { "path": ["pain_scale"],            "code": "WRONG_TYPE" },
      { "path": ["mechanism"],             "code": "NOT_IN_OPTIONS" },
      { "path": ["rom_log",0,"col_b"],     "code": "WRONG_TYPE" }
    ]
  }
}
Why it matters
The client receives every violation in one response — no fix-one-discover-another loop — and each path points at the exact offending cell, including [fieldKey, rowIndex, columnKey] inside a repeaterTable.
Aggregate · Precondition chain

Where the new check slots in.

ChartNote.saveDraft already owned a chain of invariants. Value validation lands after the cheap, surgical checks — identity, state, concurrency, key-existence — so a stale version or a typo still short-circuits before any per-field Zod work runs.

Self-defending aggregate
Validation lives inside ChartNote.saveDraft, not in the handler. Cost: the aggregate now takes templateContent: TemplateContentV2 instead of a string array. Benefit: no future caller can bypass the invariant by forgetting to call a helper.

Always valid
null for any field is accepted — drafts are incomplete by design (required-ness is the future signature-gate's problem, not this PR's). On a collection-typed field, null clears the collection; clients must send [] to keep-as-empty. Documented foot-gun.
API · Error envelope

Structured errors, one envelope.

A fourth branch in Hono's app.onError forwards the per-leaf errors[] array. The generic DomainError branch would have dropped it.

HTTP 422 Response · application/json 
{
  "error": {
    "code": "FIELD_VALUE_VALIDATION_ERROR",
    "message": "3 field value(s) failed validation",
    "errors": [
      {
        "path":    ["rom_log", 2, "col_b"],
        "code":    "WRONG_TYPE",
        "message": "expected string, got number"
      }
    ]
  }
}
code Discriminator for the client. Distinguishes FIELD_VALUE_VALIDATION_ERROR from the still-thrown UNKNOWN_FIELD_ID.
path Mirrors ZodIssue.path. Leaves are [fieldKey]; repeater cells are [fieldKey, rowIndex, columnKey]. Verbatim, never flattened.
errors[] Every violation in one response — no fix-one-discover-another loop for the UI. The client highlights each cell directly from path.
